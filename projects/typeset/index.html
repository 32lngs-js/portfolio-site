The problem

When you're designing in the browser, there's a constant context switch: you make a change in DevTools, you like it, then you have to find that property in your CSS or HTML and manually update it. The browser is the truth about how things look. The source should follow the browser, not the other way around.

This problem compounds when working with an AI coding agent. The agent writes source. You evaluate the result in the browser. You want to make a small typography adjustment. You describe it in words, the agent guesses, you go back to the browser, iterate. The loop is slow.

What I built

TypeSet is a browser overlay that intercepts typographic changes and routes them directly to source. Adjust font size, line height, or letter spacing in the panel, and the change writes back to the right file — a CSS property, a Tailwind class, a value in a style block.

It's keyed by project path, so you can run multiple workspaces simultaneously without edits crossing. It integrates with Claude Code: when an agent is watching, browser edits route directly to the agent rather than writing to file, so the agent applies the change and maintains context on what changed and why.

Key decisions

The hardest design decision was the Tailwind case. Tailwind stores values in class names on the element (e.g. text-[15px]), not in a CSS file. Writing back to source means finding that class on the element in the HTML and replacing it with the new value. TypeSet detects whether a property is managed by Tailwind or regular CSS and handles each path differently.

The routing model was a conscious choice over a simpler design (e.g. a devtools plugin). An overlay script served via CDN means zero toolchain config. Any page, any framework, any agent. You add one script tag and it works.

The meta

This site was built with TypeSet. The typography you're reading was tuned in the browser and routed back to source. The tool is embedded in its own case study, which is either a good signal or too clever by half.
